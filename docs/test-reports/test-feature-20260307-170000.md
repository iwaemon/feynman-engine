## Per-Feature Details

### Build & Test
- **Role:** Condensed matter physics PhD student trying the library
- **What they tried:** `cargo build` and `cargo test`
- **Discoverability:** Clear instructions in README. Easy to find and follow.
- **Setup:** Compiles cleanly with zero warnings and zero errors. Only standard crate dependencies.
- **Functionality:** All 36 unit tests + 3 integration tests pass in ~0.5s. No flaky or ignored tests.
- **Friction points:**
  - No MSRV (minimum supported Rust version) specified
  - No doc-tests exist
  - `cargo run` (the binary target) just prints a version string -- undocumented
  - No CI badge or configuration visible
- **Doc suggestions:** Add `rust-version` to Cargo.toml, add doc-tests to key functions, document what `cargo run` does, add a "Prerequisites" section

### Ladder Resummation
- **Role:** Condensed matter researcher computing Tc for the attractive Hubbard model
- **What they tried:** `cargo run --example ladder_resummation`
- **Discoverability:** Command clearly documented in README and usage-ja.md. Example source is readable.
- **Setup:** Ran without issues.
- **Functionality:** Output matches README character-for-character. Tc = 0.223253 is physically reasonable for U = -2t on 8x8 lattice at half-filling. Thouless criterion crosses zero correctly near Tc.
- **Friction points:**
  - Chemical potential mu=0 (half-filling) is hardcoded with no explanation
  - No guidance on convergence parameters (lattice size, Matsubara count)
  - Bisection search range (0.01, 2.0) is unexplained
  - Units are implicit (Tc is in units of t)
- **Doc suggestions:** Explain mu=0 means half-filling, add convergence discussion, document bisection range requirements, add customization guidance for varying U/lattice

### Self-Energy Diagrams
- **Role:** Graduate student studying diagrammatic perturbation theory
- **What they tried:** `cargo run --example second_order_self_energy` and DOT redirect
- **Discoverability:** Command is documented, but JSON output is completely undocumented.
- **Setup:** Compiled and ran successfully.
- **Functionality:** Text output matches docs. DOT is valid Graphviz. JSON is valid and well-formed. However, **the redirect-to-file approach documented in README/usage-ja.md is broken:**
  - `cargo run --example second_order_self_energy > diagrams.dot 2>&1` mixes cargo build messages, text output, DOT, and JSON into one file
  - Running `dot` on the result fails with syntax errors
  - The Japanese docs have a parenthetical about manually extracting the DOT block, but it's easy to miss
- **Friction points:**
  - Mixed stdout: text + DOT + JSON all go to stdout with no way to cleanly separate them
  - Topology numbering is inconsistent between text output (per-order) and DOT output (sequential)
  - Graphviz is an undocumented external dependency
  - JSON output and schema are completely undocumented
- **Doc suggestions:** Fix redirect instructions (either restructure example output or show correct `sed` extraction), document Graphviz as prerequisite, document JSON output/schema, harmonize topology numbering, consider `--format` flag

### Library API
- **Role:** Rust developer wanting to use feynman-engine as a dependency
- **What they tried:** Read source, ran `cargo doc`
- **Discoverability:** Import paths are deeply nested with no convenience re-exports. Must write paths like `feynman_engine::models::hubbard::HubbardModel`. No prelude module. `cargo doc` builds cleanly but output is sparse.
- **Setup:** Cargo.toml is **not publishable** -- missing `description`, `license`, `repository`, `authors`, `keywords`. Would work only via git URL dependency.
- **Functionality:** API surface is complete and well-structured into logical layers. The two examples effectively demonstrate both main workflows. However:
  - `PPLadder` struct is dead code (wraps HubbardModel but actual functions don't use it)
  - `find_tc` panics on bad input instead of returning `Result`
  - No `Error` type anywhere
  - `Expr` doesn't derive `PartialEq`/`Eq`
- **Friction points:**
  - Almost zero doc comments on public types and functions
  - No crate-level `//!` documentation
  - `FeynmanExpression` lives in `diagrams::rules`, not intuitive
  - Inconsistent parameter types (e.g., `q: &[f64; 2]` vs string-based site identifiers)
- **Doc suggestions:** Add crate-level docs, create `pub use` re-exports or prelude module, add `///` doc comments to all public items, fill out Cargo.toml metadata, replace panics with `Result`, remove or flesh out `PPLadder`

## Issues Found (by severity)

1. **BROKEN: DOT redirect instructions don't work.** Both README and usage-ja.md provide commands that produce invalid DOT files. Users following the docs will get Graphviz errors.
2. **MISSING: Cargo.toml metadata prevents publishing.** No `description`, `license`, or `repository` fields.
3. **MISSING: Almost no doc comments.** Key types (`HubbardModel`, `FeynmanDiagram`, `Expr`, `Observable`, etc.) have zero documentation.
4. **MISSING: No crate-level re-exports.** Import paths require knowledge of internal file structure.
5. **MISSING: JSON output feature is undocumented.** Users won't know it exists.
6. **MISSING: Graphviz listed nowhere as prerequisite.** Visualization docs assume it's installed.
7. **DESIGN: `find_tc` panics instead of returning Result.** Not idiomatic for a library.
8. **DESIGN: `PPLadder` is dead code.** Confusing for API consumers.
9. **MINOR: No MSRV specified.** Users can't know which Rust version is required.
10. **MINOR: Topology numbering inconsistent** between text and DOT output.

## Suggestions (by impact)

1. **Fix the DOT/JSON output in `second_order_self_energy` example.** Either: (a) write DOT/JSON to files directly, (b) use stderr for text and stdout for structured output, or (c) add CLI flags like `--format dot|json|text`.
2. **Add crate-level documentation and re-exports.** A `//!` block in `lib.rs` and `pub use` of key types would dramatically improve the library consumer experience.
3. **Fill out Cargo.toml** with `description`, `license`, `repository`, `authors` for eventual publishing.
4. **Add doc comments** to all public types and functions, prioritizing the 10 most-used items.
5. **Fix the redirect instructions** in README.md and docs/usage-ja.md immediately -- this is a direct copy-paste-fail for users.
6. **Document Graphviz** as an optional dependency for visualization.
7. **Add convergence guidance** for the numerical examples (lattice size, Matsubara count effects).
8. **Replace panics with Result types** in library functions like `find_tc`.
