## Per-Feature Details

### 1. Diagram Generation (Wick's Theorem)

- **Role:** PhD student integrating the library to generate Feynman diagrams for the Hubbard model
- **What they tried:** Built the project, ran Wick module tests, ran the self-energy example, read source API
- **Discoverability:** CLAUDE.md gives a clear architectural overview and the examples are the real heroes showing concrete API usage. However, there is no README.md (CLAUDE.md is AI-facing, not human-facing), and almost no rustdoc comments on public types.
- **Setup:** `cargo build` works immediately with zero friction. Minimal dependencies, fast compilation.
- **Functionality:** Everything worked. 1st-order produces 2 topologies (Hartree + Fock), 2nd-order produces 7 from 12 contractions. Physically sensible results.
- **Friction points:**
  - Import paths are deeply nested (e.g., `feynman_engine::diagrams::generate::generate_diagrams`). No re-exports or prelude.
  - `generate_diagrams` requires `&HubbardModel` but ignores it (`_model`). Misleading API coupling.
  - `Observable` enum only has `Vacuum` and `SelfEnergy` — no vertex corrections or polarization bubble. Undocumented limitation.
  - The `weight` field (`i32`) in `classify_diagrams` output is never explained.
- **Doc suggestions:** Add README.md, add rustdoc, add prelude re-exports, document `weight` semantics, document `Observable` limitations, decouple `generate_diagrams` from `HubbardModel` or document why it's required.

### 2. Classification & Feynman Rules

- **Role:** Theoretical physicist classifying diagrams by topology and producing symbolic expressions
- **What they tried:** Ran all tests (34 unit + 3 integration, all pass), ran self-energy example, read classify/rules/expr source
- **Discoverability:** The example file serves as an effective tutorial. CLAUDE.md pipeline diagram maps well to code.
- **Setup:** All tests pass in 0.33s. Build and examples run without issues.
- **Functionality:** Classification works correctly. However, symbolic expressions have significant issues:
  - **Momentum labels are unphysical.** Propagators use independent `k1, k2, ...` with no momentum conservation enforced at vertices.
  - **`Expr::U` not in output.** Interaction vertex folded into scalar prefactor, making expressions less transparent.
  - **Loop count wrong for disconnected diagrams.** Formula `n_internal - n_vertices + 1` assumes connected graphs.
  - **Free variables outside summation.** E.g., `k4, w4` appear in expressions but are never summed over — mathematically ill-defined.
- **Friction points:**
  - `FeynmanExpression.prefactor` conflates `(-U)^order * sign * symmetry_factor` — physicist would want these separated.
  - No semantic labels for diagram types (Hartree, Fock, sunset, etc.).
  - Deep import paths with no re-exports.
- **Doc suggestions:** Add rustdoc to all public types (especially `classify_diagrams` weight, `Expr` enum variants with physics context), document known limitations of symbolic expressions, add "Known Limitations" section.

### 3. Ladder Resummation & Tc Finder

- **Role:** Condensed matter physicist finding superconducting Tc of the attractive Hubbard model
- **What they tried:** Ran ladder resummation example, verified Thouless criterion, read bethe_salpeter and matsubara source
- **Discoverability:** CLAUDE.md pipeline diagram maps directly to code. The example is the key learning artifact.
- **Setup:** Builds and runs immediately. Sub-second runtime for 8x8 lattice with 256 Matsubara frequencies.
- **Functionality:** Physics results are correct and reasonable:
  - Tc = 0.2233 for t=1, U=-2, 8x8 lattice at half-filling — consistent with mean-field estimates.
  - Thouless criterion `1 - U*chi_0 = 0.00007544` at Tc (essentially zero within tolerance).
  - Correct sign structure: negative below Tc (instability), positive above (stable).
  - PP bubble frequency mapping `m-1-n` is correct.
- **Friction points:**
  - **`find_tc` hardcodes mu=0.** Silently forces half-filling. Users cannot study doped systems without writing their own bisection loop. This is the biggest functional gap.
  - `t_range` parameter undocumented — is it `(T_low, T_high)` or `(beta_low, beta_high)`? Must read source to confirm.
  - No guidance on choosing `n_matsubara` or finite-size convergence.
  - k-grid uses [0, 2pi) rather than conventional [-pi, pi) — correct but potentially confusing.
- **Doc suggestions:** Document `find_tc` parameters and mu=0 limitation, add `mu` parameter to `find_tc` or document workaround, state that mu=0 = half-filling, add convergence guidance (lattice size, n_matsubara), add reference Tc table for common parameters.

### 4. Graphviz Visualization

- **Role:** Researcher wanting to visualize Feynman diagrams as Graphviz output
- **What they tried:** Ran self-energy example, examined DOT output, read dot.rs source
- **Discoverability:** CLAUDE.md mentions the functions exist with a one-liner. Example shows usage of `to_dot_all`. `to_dot` (single diagram) is undiscovered without reading source.
- **Setup:** Works immediately. DOT output to stdout.
- **Functionality:** DOT output is syntactically valid Graphviz. Colors and styling work as documented (blue=spin-up, red=spin-down, dashed=external). However:
  - **Label inconsistency:** `to_dot()` adds `G_0↑`/`G_0↓` edge labels; `to_dot_all()` omits them entirely. Undocumented difference.
  - **Topology numbering mismatch:** Console output restarts numbering per order; DOT uses continuous numbering across all orders.
- **Friction points:**
  - No instructions for rendering DOT output (e.g., `dot -Tpng -o diagrams.png`).
  - `to_dot()` not used in any example — users won't discover it.
  - Deep import path with no re-exports.
- **Doc suggestions:** Document both functions with signatures, fix label inconsistency or document it, add rendering instructions, add visualization section explaining color conventions and topology numbering.

## Issues Found

Ordered by severity:

1. **No README.md** — All 4 testers flagged this. The project is invisible to users on GitHub. CLAUDE.md is AI-facing, not human-facing.
2. **Symbolic expressions don't enforce momentum conservation** — Propagator variables are independent placeholders with no vertex constraints. Free variables appear outside summation bounds.
3. **`find_tc` hardcodes mu=0** — Cannot study doped systems. Undocumented limitation.
4. **No rustdoc on public API** — Almost no doc comments on structs, enums, or functions. `cargo doc` produces sparse output.
5. **`generate_diagrams` requires but ignores `&HubbardModel`** — Misleading API coupling.
6. **Loop count formula wrong for disconnected diagrams** — Assumes connected graphs.
7. **`to_dot()` vs `to_dot_all()` label inconsistency** — One adds edge labels, the other doesn't.
8. **No re-exports or prelude** — Deep import paths like `feynman_engine::diagrams::generate::generate_diagrams`.
9. **`classify_diagrams` weight semantics undocumented** — `i32` could be multiplicity, sign, or both.
10. **No rendering instructions for DOT output** — Users unfamiliar with Graphviz would be stuck.

## Suggestions

Ordered by impact:

1. **Add a README.md** with project overview, quick-start example (5-10 lines to generate diagrams and find Tc), and build instructions. Keep CLAUDE.md as supplementary AI guidance.
2. **Fix symbolic expression generation** — Enforce momentum/frequency conservation at vertices; ensure all variables appear inside their summation scope.
3. **Add `mu` parameter to `find_tc`** (or accept `ThermalParams` template) so users can study non-half-filled systems.
4. **Add rustdoc comments** to all public types and functions, especially `classify_diagrams` (weight meaning), `FeynmanDiagram` fields, `Expr` variants, and `find_tc` parameters.
5. **Add prelude or crate-root re-exports** — `pub use` key types so users can write `use feynman_engine::prelude::*`.
6. **Fix `to_dot`/`to_dot_all` inconsistency** — Either both should label edges or make it configurable.
7. **Decouple `generate_diagrams` from `HubbardModel`** — The model is unused; accept only `order` and `Observable`.
8. **Add a "Known Limitations" section** documenting: symbolic momentum conservation, `Observable` variants, lattice types, mu=0 in `find_tc`.
9. **Add DOT rendering instructions** — e.g., `cargo run --example second_order_self_energy | dot -Tpng -o diagrams.png`.
10. **Add convergence guidance** — Suggest checking lattice size and `n_matsubara` convergence for numerical results.
